\section{Төстэй системүүдийн судалгаа}

AI-д суурилсан видео контент үүсгэлтийн чиглэлээр дотоод болон гадаадад хэд хэдэн судалгааны ажил хийгдсэн байна. Олон улсын туршлагаас харвал дараах системүүд амжилттай ажиллаж байна:

\begin{itemize}
\item \textbf{InVideo AI} (АНУ, 2017 оноос): Текст prompt-оос видео, дуу хоолой, визуал нэгтгэн автоматаар контент үүсгэдэг. 16 сая гаруй медиа хөрөнгөтэй.
\item \textbf{Pictory AI} (АНУ, 2021 оноос): Скрипт болон блог постоос видео автоматаар үүсгэх, хялбар ашиглалттай платформ.
\item \textbf{Synthesia} (Их Британи, 2017 оноос): Хиймэл аватар ашиглан корпорейт видео бүтээх, 70+ хэлд дэмжигдсэн.
\item \textbf{Runway Gen-3} (АНУ): Мэргэжлийн түвшний AI видео үүсгэлт, кино үйлдвэрлэлд хэрэглэгддэг.
\end{itemize}

Монгол улсын хувьд ижил төстэй автомат видео контент үүсгэлтийн систем одоогоор байхгүй байна. Б.Мөнхбаяр нарын 2024 оны судалгааны ажилд AI ашиглан текстийг дуу хоолойд хөрвүүлэх системийн шинжилгээ хийгдсэн байна.

\section{Ашиглах AI загваруудын судалгаа}

\subsection{AI загваруудын харьцуулалт ба сонголтын үндэслэл}

Энэхүү систем нь богино хэмжээний AI видео автоматаар үүсгэдэг тул 3 үндсэн төрлийн загвар сонгох шаардлагатай. Үүнд:

\begin{itemize}
\item \textbf{Script/Text generation model} --- видеоны гарчиг, scene, narration, image prompt үүсгэх
\item \textbf{Image generation model} --- scene бүрийн зураг үүсгэх
\item \textbf{Voice generation model} --- narration-ийг дуу болгон хувиргах
\end{itemize}

Сонголт хийхдээ зөвхөн чанарыг бус, хурд, өртөг, интеграцийн хялбар байдал, production readiness, multilingual support зэрэг шалгуурыг хамтад нь авч үзэх нь зүйтэй.

\subsection{Script generation model-уудын харьцуулалт}

Энэ хэсэгт дараах 3 сонголтыг авч үзэв:

\begin{itemize}
\item Groq + Llama 3.3 70B Versatile
\item Claude Sonnet family
\item Gemini family
\end{itemize}

Groq-ийн llama-3.3-70b-versatile нь 131,072 token context, 32,768 max output, мөн Groq pricing хуудсанд 394 TPS орчим throughput, \$0.59 / 1M input token, \$0.79 / 1M output token гэж дурдагдсан байна. Энэ нь өндөр хурдтай, харьцангуй хямд inference хийхэд давуу.

Claude Sonnet family нь Anthropic-ийн models overview-д fast latency, 200K context window, Sonnet 4.5/4.6 ангиллуудад \$3 / 1M input, \$15 / 1M output token үнэтэй гэж байна. Claude нь reasoning, analysis, multilingual support дээр хүчтэй гэж албан ёсоор тайлбарлагдсан.

Gemini Developer API pricing хуудсанд Gemini text model-уудын үнэ, мөн preview model-ууд нь stable хувилбараас илүү хязгаарлагдмал rate limit-тэй, өөрчлөгдөх боломжтой гэж анхааруулсан. Vertex AI models хуудсанд Gemini 3.1 Pro нь complex multimodal understanding-д зориулагдсан гэж тэмдэглэсэн. \subsubsection{Script model сонголтын дүгнэлт}

Манай системийн хувьд script generation нь олон scene, олон request, богино хугацаанд ажиллах тул Groq + Llama 3.3 70B хамгийн тохиромжтой гэж үзэв. Гол үндэслэл нь:

\begin{itemize}
\item өртөг Claude-аас мэдэгдэхүйц бага,
\item throughput өндөр,
\item scene breakdown, narration, image prompt зэрэг structured output-д хангалттай,
\item API integration энгийн.
\end{itemize}

Хэрэв future version дээр илүү нарийн reasoning, өндөр чанартай creative script хэрэгтэй бол Claude Sonnet-ийг fallback буюу premium mode болгон ашиглаж болно.

Иймээс script generation-д сонгосон модель: \textbf{Groq + Llama 3.3 70B Versatile}.

\subsection{Image generation model-уудын харьцуулалт}

Image generation хэсэгт дараах сонголтуудыг авч үзэв:

\begin{itemize}
\item Nano Banana / Gemini image generation
\item FLUX (Black Forest Labs)
\item Imagen 4
\end{itemize}

Google Vertex AI documentation-д Gemini image generation нь одоогоор preview бөгөөд production-ready image generation хэрэгтэй бол Imagen ашиглахыг зөвлөсөн байна. Мөн gemini-3.1-flash-image-preview, gemini-3-pro-image-preview, gemini-2.5-flash-image зэрэг сонголтууд listed байна. Gemini 3 Pro Image нь complex, multi-turn image generation ба editing-д хамгийн сайн гэж Google тайлбарласан.

Google-ийн Imagen documentation-д Imagen 4 нь хамгийн сүүлийн image generation line гэж тэмдэглэгдсэн бөгөөд Vertex AI дээр generate, edit, upscale хийх боломжтой. Энэ нь enterprise/production орчинд илүү тогтвортой сонголт болохыг илтгэнэ.

Black Forest Labs-ийн FLUX API documentation-д FLUX.2 pricing нь megapixel-based буюу resolution өсөх тусам өртөг өсдөг, мөн production-scale API санал болгодог гэж дурдсан. Энэ нь зургийн хэмжээ, чанарыг өртөгтэй шууд уялдуулж удирдах давуу талтай.

Gemini pricing хуудсанд image output нь token-based үнэтэй бөгөөд 1024×1024 хэмжээтэй image ойролцоогоор \$0.067, 2048×2048 image ойролцоогоор \$0.101 гэж дурдсан. Энэ нь image cost-ийг урьдчилан тооцоход ашигтай.

\subsubsection{Image model сонголтын дүгнэлт}

Энд хоёр өөр өнцөг байна. Хэрэв гол зорилго нь production system бол Google өөрөө ``preview image generation-аас илүү production-ready хэрэгцээнд Imagen ашигла'' гэж зөвлөж байгаа тул Imagen 4 хамгийн зөв сонголт болно.

Хэрэв гол зорилго нь creative edit/regenerate workflow бол Nano Banana буюу Gemini image generation нь multi-turn editing дээр илүү эвтэйхэн. Гэхдээ preview model тул дипломын ажил дээр ``туршилтын түвшинд сайн, production талд эрсдэлтэй'' гэж тайлбарлах нь зөв.

FLUX нь image quality ба scale талдаа хүчтэй өрсөлдөгч бөгөөд resolution-based pricing нь олон зураг гаргах pipeline-д cost planning хийхэд сайн. Гэхдээ танай workflow scene prompt $\rightarrow$ regenerate $\rightarrow$ edit гэсэн хэлбэртэй бол Google-ийн image stack илүү нэгдмэл мэдрэмж өгнө.

\subsubsection{Эцсийн сонголт}

Танай систем short-form video production pipeline тул би энэ байдлаар санал болгож байна:

\begin{itemize}
\item Хэрэв диплом дээр ``production-ready'' гэдгийг хүчтэй хамгаалах бол: Imagen 4
\item Хэрэв диплом дээр ``creative editing + prompt iteration'' гэдгийг онцлох бол: Nano Banana / Gemini image
\item Хэрэв cost-control + API flexibility-ийг онцлох бол: FLUX
\end{itemize}

Манай төсөлд: Үндсэн image model: \textbf{Imagen 4}, туршилтын/итерацийн хувилбар: \textbf{Nano Banana}. Учир нь Imagen нь stable талдаа, Nano Banana нь experimental creative side дээр давуу гэж тайлбарлаж болно.

\section{Ашиглах хөгжүүлэлтийн технологиудын судлагаа}

Энэхүү систем нь AI-д суурилсан богино хэмжээний видео контентыг автоматаар үүсгэх веб систем тул frontend, backend, өгөгдлийн сан, ORM, API архитектур, мөн бүх түвшинд type safety хангах технологийн сонголт чухал юм. Манай системд frontend-д React, backend-д Node.js ба Express, өгөгдлийн санд PostgreSQL, ORM түвшинд Sequelize, API харилцаанд REST API, харин бүх шатанд TypeScript ашиглахаар сонгосон. Эдгээр технологиуд нь хоорондоо сайн нийцдэг, өргөн хэрэглэгддэг, хөгжүүлэлтийн болон засвар үйлчилгээний зардлыг бууруулах боломжтой стек бүрдүүлдэг.

\subsection{Frontend технологи: React}

React нь хэрэглэгчийн интерфэйсийг component-based байдлаар хөгжүүлэх JavaScript сан бөгөөд интерфэйсийг жижиг, дахин ашиглаж болох хэсгүүдэд хуваан боловсруулах боломж олгодог. React-ийн албан ёсны баримтад ``user interface-ийг components-оос бүрдүүлэн бүтээх'' зарчим нь гол онцлог гэж тайлбарласан байдаг. Энэ нь манай системийн video generation form, scene editor, preview panel, history, dashboard зэрэг олон хэсэгтэй интерфэйс боловсруулахад тохиромжтой.

React-ийг сонгосон шалтгаан нь:

\begin{itemize}
\item component-based бүтэцтэй тул интерфэйсийг модульчилж хөгжүүлэхэд хялбар,
\item state-driven rendering ашигладаг тул video generation progress, scene data, preview зэрэг байнга өөрчлөгдөх UI-д тохиромжтой,
\item өргөн community болон ecosystem-тэй тул хөгжүүлэлтийн хурд өндөр,
\item TypeScript-тэй сайн хосолдог.
\end{itemize}

Манай системд хэрэглэгч олон scene бүхий видео үүсгэх, scene тус бүрийн narration, зураг, subtitle зэргийг засварлах шаардлагатай тул динамик, interactive интерфэйс хэрэгтэй. Энэ шаардлагад React оновчтой сонголт болно.

\subsection{Backend технологи: Node.js}

Node.js нь asynchronous, event-driven JavaScript runtime бөгөөд scalable network application хөгжүүлэхэд зориулагдсан. Албан ёсны ``About Node.js'' хуудсанд Node.js нь non-blocking I/O загварт тулгуурладаг бөгөөд олон холболтыг зэрэг боловсруулахад тохиромжтой гэж тайлбарласан. Манай системд AI model API дуудлага, database access, media processing request, file upload, progress update зэрэг олон I/O-bound үйлдэл нэгэн зэрэг явагддаг тул Node.js маш тохиромжтой.

Node.js-ийг сонгосон үндэслэл нь:

\begin{itemize}
\item asynchronous event loop ашигладаг тул API orchestration хийхэд сайн,
\item AI inference, database, storage зэрэг олон service-тэй зэрэг харилцахад тохиромжтой,
\item TypeScript болон Express-тэй сайн уялддаг,
\item JavaScript/TypeScript хэл нэг байх тул frontend-backend хооронд хөгжүүлэлт илүү жигд болдог.
\end{itemize}

AI видео системийн backend нь CPU-heavy video rendering-ийг өөрөө хийхээс илүү гадаад AI API, FFmpeg, DB, cloud storage зэргийг уялдуулдаг ``orchestration layer'' байдлаар ажиллах учир Node.js-ийн I/O-oriented архитектур оновчтой гэж үзэв.

\subsection{Web framework: Express}

Express нь Node.js дээр суурилсан fast, unopinionated, minimalist web framework юм. Express-ийн албан ёсны сайт үүнийг энгийн, уян хатан, web application болон API хөгжүүлэхэд зориулагдсан framework гэж тодорхойлдог. Манай системд user authentication, video generation endpoint, scene update endpoint, media retrieval, admin route зэрэг олон REST endpoint боловсруулах шаардлагатай тул Express тохиромжтой.

Express-ийг сонгосон шалтгаан нь:

\begin{itemize}
\item routing, middleware architecture нь API хөгжүүлэлтэд хялбар,
\item validation, authentication, logging, error handling-ийг middleware хэлбэрээр зохион байгуулах боломжтой,
\item Node.js экосистемтэй маш сайн нийцдэг,
\item REST API сервер байгуулахад өргөн хэрэглэгддэг.
\end{itemize}

Манай системийн backend нь том хэмжээний monolithic framework шаардахгүй, харин тодорхой API endpoint-уудтай, AI pipeline-тэй холбогдох уян хатан сервер хэрэгтэй тул Express илүү зохимжтой.

\subsection{Өгөгдлийн сан: PostgreSQL}

PostgreSQL нь нээлттэй эхийн object-relational database system бөгөөд найдвартай байдал, data integrity, extensibility, performance-аараа өргөн танигдсан. PostgreSQL-ийн албан ёсны ``About'' хуудсанд энэ систем нь SQL-ийг өргөтгөн ашигладаг, төвөгтэй өгөгдлийн ачааллыг аюулгүй хадгалж, өсгөхөд зориулагдсан гэж тайлбарласан. Мөн PostgreSQL нь ACID-compliant, proven architecture-тэй, өргөтгөх боломж өндөртэй гэж дурдсан байдаг.

PostgreSQL-ийг сонгосон шалтгаан нь:

\begin{itemize}
\item хэрэглэгч, видео, scene, asset, generation status зэрэг структуртай өгөгдөл хадгалахад тохиромжтой,
\item relational холбоосуудыг сайн дэмждэг,
\item transaction support сайн тул video generation workflow-ийн consistency хадгалахад ашигтай,
\item open-source бөгөөд production орчинд өргөн ашиглагддаг.
\end{itemize}

Манай системд дараах төрлийн өгөгдлүүд хадгалагдана:

\begin{itemize}
\item хэрэглэгчийн мэдээлэл,
\item үүсгэсэн төслүүд,
\item scene бүрийн текст, зураг, аудио URL,
\item video generation-ийн төлөв,
\item subscription болон usage мэдээлэл.
\end{itemize}

Эдгээр нь хоорондоо relational холбоос ихтэй тул PostgreSQL хамгийн зөв сонголт гэж үзэв.

\subsection{ORM технологи: Sequelize}

Sequelize нь Node.js болон TypeScript-д зориулсан ORM бөгөөд Postgres, MySQL, MariaDB, SQLite, SQL Server зэрэг олон өгөгдлийн санг дэмждэг. Sequelize-ийн албан ёсны хуудаст modern TypeScript and Node.js ORM гэж тодорхойлж, transaction support, relations, eager/lazy loading, read replication зэрэг боломжуудыг онцолсон байна.

Sequelize-ийг сонгосон үндэслэл нь:

\begin{itemize}
\item PostgreSQL-тэй шууд ажиллах боломжтой,
\item model definition, CRUD, associations-ийг abstraction түвшинд хялбар болгодог,
\item hasMany, belongsTo зэрэг холбоосуудыг илэрхийлэхэд тохиромжтой,
\item transaction ашиглан өгөгдлийн бүрэн бүтэн байдлыг хангах боломжтой,
\item TypeScript-тэй хамт ашиглаж model type, DTO type, service layer type-ийг цэгцтэй зохион байгуулах боломжтой.
\end{itemize}

Манай системд жишээ нь:

\begin{itemize}
\item нэг хэрэглэгч олон видео үүсгэнэ,
\item нэг видео олон scene-тэй байна,
\item нэг scene олон asset-тай байж болно
\end{itemize}

гэсэн relational бүтэц шаардлагатай. Sequelize-ийн association support нь ийм бүтэцтэй системд тохиромжтой.

Гэхдээ нэг чухал зүйл нь Sequelize-ийн сайт дээр v6 нь stable, харин v7 нь alpha гэж тодорхой заасан байгаа. Иймээс production болон дипломын төслийн практик хэрэгжилтэд stable хувилбар ашиглах нь зүйтэй гэж үзэж болно.

\subsection{API архитектур: REST API}

Манай системийн frontend ба backend хоорондын харилцаанд REST API ашиглана. REST нь HTTP дээр суурилсан resource-oriented харилцааны хэв маяг бөгөөд HTTP өөрөө client-server, stateless протокол гэдгийг MDN тайлбарласан. Мөн HTTP methods буюу GET, POST, PUT/PATCH, DELETE нь тодорхой төрлийн үйлдлийг илэрхийлэхэд ашиглагддаг. Энэ шинж нь RESTful API дизайн хийх үндэс болдог.

REST API-ийг сонгосон шалтгаан нь:

\begin{itemize}
\item frontend ба backend-г тодорхой заагтай холбодог,
\item resource-based endpoint дизайн хийхэд ойлгомжтой,
\item authentication, CRUD, media metadata, generation status зэрэг өгөгдөл дамжуулахад тохиромжтой,
\item React frontend-тэй уялдуулахад энгийн.
\end{itemize}

Манай системийн жишээ endpoint-ууд:

\begin{itemize}
\item \texttt{POST /videos} --- шинэ видео үүсгэх
\item \texttt{GET /videos/:id} --- видеоны мэдээлэл авах
\item \texttt{GET /videos/:id/scenes} --- scene жагсаалт авах
\item \texttt{PATCH /scenes/:id} --- scene засварлах
\item \texttt{DELETE /videos/:id} --- видео устгах
\end{itemize}

гэх мэт resource-oriented бүтэцтэй байж болно. Энэ нь системийг өргөтгөхөд хялбар болгоно. REST загвар нь ялангуяа frontend, mobile client, admin panel зэрэг олон хэрэглэгч талтай бол илүү тохиромжтой.

\subsection{Бүх түвшний хэл: TypeScript}

TypeScript нь JavaScript дээр type syntax нэмж, static type checking хийх боломж олгодог. TypeScript Handbook-д түүний зорилгыг ``JavaScript program-д зориулсан static typechecker'' гэж тодорхойлсон. Мөн TypeScript нь editor integration, type inference, алдааг эрт илрүүлэх боломжийг сайжруулдаг гэж албан ёсны сайт тайлбарладаг.

Манай системд TypeScript-ийг frontend, backend, ORM model, API contract бүх түвшинд ашиглах нь дараах давуу талтай:

\begin{itemize}
\item compile хийхээс өмнө type error илрүүлнэ,
\item DTO, request, response, scene object, video metadata зэрэг өгөгдлийн бүтцийг тодорхой болгоно,
\item refactor хийхэд илүү аюулгүй,
\item frontend ба backend хооронд shared type ашиглах боломжтой,
\item багийн хөгжүүлэлтэд code readability ба maintainability сайжирна.
\end{itemize}

AI video system-д scene, narration, subtitle, asset, prompt, user quota, generation job зэрэг олон object байдаг. Эдгээрийн бүтэц байнга дамждаг тул type safety маш чухал. JavaScript дээр ийм систем хөгжүүлэхэд runtime дээр илрэх алдааны магадлал өндөр байдаг бол TypeScript энэ эрсдэлийг бууруулна.

\subsection{Видео боловсруулалт: FFmpeg}

FFmpeg нь мультимедиа тайлах, кодлох, нэгтгэх, хөрвүүлэх нэгдмэл хэрэгсэл бөгөөд олон төрлийн аудио, видео, зургийн форматыг дэмжсэн нээлттэй эхийн платформ юм. Энэ нь манай системд бүх scene-ийн зураг, narration, subtitle, background music-ийг нэгтгэн эцсийн MP4 видео гаргах гол хэрэгсэл болдог.

FFmpeg-ийг сонгосон шалтгаан нь:

\begin{itemize}
\item олон төрлийн codec, container формат дэмждэг (MP4, MOV, WebM, MP3, WAV, AAC гэх мэт),
\item командын мөрөөс програмчлалаар удирдах боломжтой,
\item server-side rendering хийхэд тохиромжтой,
\item Ken Burns effect, fade in/out, crossfade зэрэг transition effect дэмждэг,
\item subtitle (SRT, ASS) шууд видеонд бүлгэн оруулах боломжтой,
\item олон scene-ийг нэгтгэх, тус бүрээр дахин render хийх боломжтой.
\end{itemize}

\subsubsection{Transition ба motion effect}

Богино хэмжээний видео контентыг сонирхолтой болгоход transition болон motion effect чухал үүрэгтэй. Манай системд FFmpeg ашиглан дараах эффектүүдийг хэрэгжүүлнэ:

\begin{itemize}
\item \textbf{Ken Burns effect} --- статик зураг дээр аажмаар zoom in/out хийх замаар динамик харагдац үүсгэх. Энэ нь scene бүрт амьд хөдөлгөөн нэмж, хэрэглэгчийн анхаарлыг барина.
\item \textbf{Crossfade transition} --- хоёр scene-ийн хооронд гөлгөр шилжилт хийх. Нэг scene аажмаар бүдгэрч, дараагийнх нь гарч ирнэ.
\item \textbf{Fade in/out} --- видеоны эхэн ба төгсгөлд хар дэлгэцээс/рүү шилжих эффект.
\item \textbf{Pan motion} --- зургийг зүүн-баруун эсвэл дээш-доош хөдөлгөж parallax эффект үүсгэх.
\item \textbf{Slide transition} --- scene-ийг хажуу тийш гулсуулах байдлаар солих.
\end{itemize}

\subsubsection{Subtitle ба background music нэгтгэлт}

FFmpeg-ийн \texttt{filter\_complex} функц нь олон input stream-ийг нэгтгэх боломжийг олгодог. Манай системд:

\begin{itemize}
\item Narration audio-г scene timing-тэй уялдуулан давхарлах,
\item Background music-ийг narration-ы доор намхан volume-оор тоглуулах,
\item Subtitle-ийг видеонд burn-in хэлбэрээр оруулж, ямар ч төхөөрөмжид харагдахаар болгох,
\item Audio ducking ашиглан narration ярих үед хөгжмийн дууг автоматаар намсгах
\end{itemize}

зэрэг үйлдлүүдийг гүйцэтгэнэ.

\subsubsection{Хэсэгчилсэн re-render}

Манай системийн нэг гол давуу тал нь хэрэглэгч зөвхөн нэг scene-ийг засварлахад бүх видеог биш, тухайн scene-ийг ганцыг л дахин render хийдэг. Үүнийг FFmpeg-ийн segment-based encoding ашиглан хэрэгжүүлснээр rendering хугацаа болон server resource-ийн хэрэглээг эрс багасгана.
\subsection{Технологийн нэгдсэн дүгнэлт}

Энэхүү системд сонгосон хөгжүүлэлтийн стек нь:

\begin{itemize}
\item React --- динамик, component-based frontend боловсруулах,
\item Node.js --- asynchronous, scalable backend orchestration хийх,
\item Express --- REST API сервер байгуулах,
\item PostgreSQL --- relational, найдвартай өгөгдөл хадгалах,
\item Sequelize --- DB model ба холбоосуудыг abstraction түвшинд удирдах,
\item REST API --- frontend/backend харилцааг resource-based байдлаар зохион байгуулах,
\item TypeScript --- бүх түвшинд type safety хангах
\end{itemize}

гэсэн үүрэгтэйгээр сонгогдсон. Эдгээр технологиуд нь хоорондоо сайн нийцдэг бөгөөд AI-д суурилсан веб систем хөгжүүлэхэд practical, scalable, maintainable шийдэл болж байна.

\section{Бүлгийн дүгнэлт}

Энэ бүлэгт AI-д суурилсан богино хэмжээний видео контент автоматаар үүсгэх системд ашиглах загварууд болон хөгжүүлэлтийн технологиудыг судалж, харьцуулан дүгнэв. Судалгааны үр дүнд script generation хэсэгт хурд, өртгийн хувьд Groq + Llama 3.3 70B, image generation хэсэгт тогтвортой байдал, чанарын хувьд Imagen 4, voice generation хэсэгт naturalness болон олон хэлний дэмжлэгийн хувьд ElevenLabs-ийг ашиглах нь илүү тохиромжтой гэж үзэв. Эдгээр сонголт нь системийн үндсэн шаардлага болох чанартай контент богино хугацаанд үүсгэх боломжийг хангаж байна.

Мөн системийн хөгжүүлэлтийн технологийн судалгаагаар frontend хэсэгт React, backend хэсэгт Node.js ба Express, өгөгдлийн санд PostgreSQL, ORM түвшинд Sequelize, харилцааны түвшинд REST API, бүх шатанд TypeScript ашиглах нь оновчтой гэж дүгнэв. Энэ стек нь хоорондоо сайн уялдаатай, хөгжүүлэлт хийхэд хялбар, өргөтгөх боломжтой бөгөөд AI-д суурилсан веб систем боловсруулахад тохиромжтой байна.

Иймээс энэхүү бүлгийн судалгааны үр дүнд манай системийг хэрэгжүүлэхэд шаардлагатай загвар болон технологийн үндсэн сонголтуудыг тодорхойлж, цаашдын системийн шаардлага тодорхойлох болон хөгжүүлэлтийн ажлын суурь үндэслэл бүрдсэн гэж үзэж байна.
