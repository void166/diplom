\section{Системийн архитектур боловсруулах}

Хөгжүүлж буй системийн архитектурыг гурван давхаргат (3-tier) архитектурын зарчмын дагуу бүтээв. Энэхүү бүтэц нь интерфэйсийн давхарга (Presentation Layer), бизнес логикийн давхарга (Application Layer), өгөгдлийн давхарга (Data Layer) гэсэн гурван үндсэн хэсгээс бүрдэх бөгөөд давхарга бүр өөр өөр үүрэгтэй, харилцан тодорхой интерфэйсээр уялдаатай ажиллана.

\subsection{Системийн ерөнхий архитектур}

Системийн интерфэйсийн давхарга нь React дээр суурилсан Single-Page Application бөгөөд хэрэглэгч browser-ээр дамжуулан системд хандана. Бизнес логикийн давхарга нь Node.js + Express + TypeScript-аар хөгжүүлсэн RESTful API сервер бөгөөд видео үүсгэлтийн урсгал, AI үйлчилгээний оркестрэйшн, баталгаажуулалт болон бүртгэлийн логикийг хариуцна. Өгөгдлийн давхарга нь PostgreSQL реляцийн өгөгдлийн сан болон AWS S3, Cloudinary үүлэн хадгалалтаас бүрдэнэ. Гадаад AI үйлчилгээнүүд (Groq, Imagen, ElevenLabs) болон видео боловсруулалтын FFmpeg worker нь системийн дотоод pipeline-аас тусдаа орчинд ажиллана.

Энэхүү давхаргат загвар нь дараах давуу тал олгоно. Нэгдүгээрт, frontend болон backend нь тус тусдаа deploy хийгдэх боломжтой тул scaling уян хатан болно. Хоёрдугаарт, AI үйлчилгээнүүдийг солих эсвэл нэмэх үед зөвхөн бизнес логикийн давхаргад өөрчлөлт хийхэд хангалттай. Гуравдугаарт, frontend болон backend хооронд REST API стандартаар уялдсан тул mobile эсвэл desktop client-ийг ирээдүйд нэмж хөгжүүлэх боломж нээлттэй байна.

\subsection{Backend дотоод архитектур}

Backend серверийг Layered Architecture буюу давхаргат архитектурын зарчмын дагуу зохион байгуулсан. Гол давхарганууд нь дараах байдлаар үүрэгчилэгдсэн:

\begin{itemize}
  \item \textbf{Routes давхарга} --- HTTP хүсэлтийг хүлээн авч, тухайн endpoint-д харгалзах controller-руу чиглүүлнэ. Express router-ыг ашиглана.
  \item \textbf{Controller давхарга} --- хүсэлт болон хариуны форматыг боловсруулж, validation хийн, харгалзах service-ийг дуудна. Бизнес логик агуулахгүй.
  \item \textbf{Service давхарга} --- гол бизнес логик, AI service-тэй харилцах оркестрэйшн, transaction удирдлага зэрэг ажилладаг гол давхарга.
  \item \textbf{Repository давхарга} --- Sequelize ORM-ийг ашиглан өгөгдлийн санд CRUD үйлдэл гүйцэтгэнэ. Бусад давхаргад өгөгдлийн сангийн дэлгэрэнгүй мэдэгдэхгүй.
  \item \textbf{Middleware давхарга} --- JWT баталгаажуулалт, error handling, rate limiting, logging, request validation зэрэг хөндлөнгийн логикийг агуулна.
  \item \textbf{Worker давхарга} --- FFmpeg видео угсралт зэрэг хүнд ажлыг queue-ээр нь зэрэгцээ ажиллуулах хариуцлагатай.
\end{itemize}

Энэ бүтэц нь Single Responsibility Principle-ийг хадгалж, кодын засвар үйлчилгээний өртгийг бууруулдаг. Жишээ нь, өгөгдлийн санг ирээдүйд PostgreSQL-ээс өөр санд шилжүүлэхэд зөвхөн Repository давхаргад өөрчлөлт хийхэд хангалттай.

\subsection{Frontend архитектур}

Frontend интерфэйсийг React 18 + TypeScript ашиглан Single-Page Application хэлбэрээр хөгжүүлэв. Component-based архитектур ашиглаж, дахин ашиглах боломжтой жижиг компонентоос нийлмэл хуудас нэгтгэнэ. State management-д React Context API ба TanStack Query (React Query) ашигласан. TanStack Query нь backend-аас ирэх өгөгдлийн caching, refetching, optimistic update зэргийг хариуцна.

Routing-д React Router v6 ашиглаж, хуудас бүрд lazy loading хийсэн. Энэ нь анхны ачааллалтын хугацааг богиносгож, хэрэглэгчийн туршлагыг сайжруулсан. Стилийг Tailwind CSS-ээр хийсэн бөгөөд утилитын класс ашиглах нь стиль бичих хурдыг нэмэгдүүлсэн.

\section{Өгөгдлийн сангийн бүтэц зохиох}

Өгөгдлийн сангийн зохиомжид PostgreSQL 15 реляцийн өгөгдлийн санг сонгосон. Энэхүү хэсэгт өгөгдлийн сангийн логик загвар, нормчилол болон Entity-Relationship буюу ER загварчлалыг авч үзнэ.

\subsection{Entity тодорхойлолт}

Системийн өгөгдлийн санд хадгалагдах үндсэн объектуудыг дараах байдлаар тодорхойлсон:

\begin{itemize}
  \item \textbf{Хэрэглэгч (User)} --- системд бүртгэлтэй хэрэглэгчийн үндсэн мэдээлэл: имэйл, нууц үгийн хэш, нэр.
  \item \textbf{Видео (Video)} --- нэг бүхэл видеоны мета мэдээлэл, үүсгэлтийн төлөв, зурагны хэв маяг, дуут замын тохиргоо болон эцсийн URL.
  \item \textbf{Scene} --- видеоны нэг хэсэг бүрийн narration текст, image prompt, аудио, үргэлжлэх хугацаа, дуу хоолойн тохиргоо.
  \item \textbf{Төсөл (Project)} --- хэрэглэгчийн видеонуудыг бүлэглэх хавтас бүтэц. Нэг видео нэг төсөлд харьяалагдаж болно.
  \item \textbf{Subscription} --- хэрэглэгчийн тарифын төлөв, сарын видео үүсгэх хязгаарлалт, дуусах хугацаа.
  \item \textbf{Админ (Admin)} --- системийг удирдах тусдаа бүртгэлтэй хэрэглэгч. Энгийн хэрэглэгчтэй тусдаа хүснэгтэд хадгалагдана.
\end{itemize}

Эдгээр объектуудын хооронд дараах гол холбоосууд тогтоогдсон: нэг хэрэглэгч олон видеотой (1:N), нэг хэрэглэгч олон төсөлтэй (1:N), нэг төсөл олон видеотой (1:N), нэг видео олон scene-тэй (1:N), нэг хэрэглэгч нэг subscription-тай (1:1).

\subsection{Authentication ба JWT хэрэгжүүлэлт}

Баталгаажуулалтыг JSON Web Token (JWT) дээр суурилуулсан. Хэрэглэгч нэвтрэх бүрд хоёр төрлийн токен олгоно:

\begin{enumerate}
\item \textbf{Access token} --- 15 минутын хугацаатай, бүх API хүсэлтэд \code{Authorization} header-аар дамжуулна.
\item \textbf{Refresh token} --- 7 хоногийн хугацаатай, HttpOnly cookie-д хадгалагдаж, access token дуусах үед шинэ access token авахад ашиглагдана.
\end{enumerate}

Энэ хослол нь хоёр давуу талтай. Access token хулгайлагдсан тохиолдолд 15 минутаас илүү ашиглах боломжгүй. Refresh token нь HttpOnly cookie-д хадгалагдсан тул JavaScript-ээс хандаж болохгүй учир XSS халдлагаас хамгаалагдана.

Нууц үгийг bcrypt-ийн 12 salt round ашиглан хэшлэн хадгална. Энэ нь brute-force халдлагад тэсвэртэй болохын зэрэгцээ нэг нууц үг хэшлэхэд орчим 250--300ms зарцуулдаг учир CPU-д хэт ачаалал өгөхгүй.

\section{AI service-үүдтэй холболт хийх}

Системийн гол үнэ цэнэ нь хиймэл оюуны үйлчилгээнүүдтэй уялдаа холбоотой ажиллахад оршино. Бүх AI үйлчилгээний интеграцыг service давхаргад тусгайлан тусгаарласан тул ирээдүйд үйлчилгээний нийлүүлэгчийг солих эсвэл fallback нэмэхэд хялбар байх бүтцийг хадгалсан.

\subsection{Groq + Llama 3.3 70B (скрипт үүсгэлт)}

Скрипт үүсгэлтэд Groq cloud-ыг ашиглан Llama 3.3 70B Versatile загварыг дуудна. Хэрэглэгчийн оруулсан сэдэв, жанр, урт, хэл зэрэг параметрийг нарийн боловсруулсан system prompt-ын хамт илгээж, бүтэцлэгдсэн JSON хариулт авна. JSON хариулт нь scene array байх бөгөөд scene бүр \code{narration}, \code{image\_prompt}, \code{duration\_seconds} талбартай.

Хариулсан JSON хүчинтэй эсэхийг zod schema-аар шалгаж, хүчинтэй биш бол retry хүсэлт явуулна. Гурван удаагийн retry-аар хариу авч чадахгүй бол алдаа болж видео үүсгэлт зогсох ба хэрэглэгчид мэдэгдэл явна. Дундаж хариу авах хугацаа 6 scene-тэй видеонд 2--4 секунд болсон.

\subsection{Imagen 4 / Nano Banana (зураг үүсгэлт)}

Зураг үүсгэлтэд үндсэн загвар болгон Google Vertex AI дээрх Imagen 4 загварыг ашигласан. Scene бүрийн image prompt-ыг систем нь хэрэглэгчийн өгсөн жанр, өнгөний хэв маяг, видеоны хэлбэрт нийцүүлж нэмэлт стилийн заавартай хослуулж илгээдэг. Жишээ нь, TikTok видеонд \textit{``vertical format, vibrant colors, dynamic''} гэх мэт нэмэлт мэдээлэл нэмэгддэг.

Үүсгэгдсэн зургийг Cloudinary дээр upload хийсний дараа \code{scenes.image\_url} талбарт URL хадгалагдана. Дундаж хугацаа 1 зураг үүсгэхэд 4--7 секунд орсон.

\subsection{ElevenLabs (дуу хоолой синтез)}

Дуу хоолой синтезд ElevenLabs API ашигласан. Хурдны зориулалттай Flash v2.5 загвар ($\sim$75ms latency) ба чанарын зориулалттай Multilingual v2 загвар хосолж ашиглах загварыг хэрэгжүүлсэн.

ElevenLabs нь Монгол хэлийг шууд дэмждэггүй боловч Multilingual v2 загвар нь хэлийг автоматаар таних чадвартай тул Монгол текстийг кирилл бичгээр дамжуулахад зөв уншдаг. \code{voice\_settings.stability} болон \code{voice\_settings.similarity\_boost} параметрүүдийг тохируулж Монгол дуудлагын чанарыг сайжруулсан. Дундаж 30 секундын narration-д 3--5 секунд зарцуулагдсан.

\subsection{FFmpeg видео угсралтын pipeline}

Зураг ба audio-г нэгтгэж эцсийн MP4 видео үүсгэх ажлыг FFmpeg хариуцна. Гол алхмуудыг доор тоочив:

\begin{enumerate}
\item Scene бүрийн зургаас тухайн scene-ийн audio-той тохирсон урттай видеогийн фрагмент үүсгэнэ (\code{filter\_complex} дотор \code{loop} ашиглана).
\item Subtitle-уудыг SRT файл болгон гаргаж \code{drawtext} эсвэл \code{subtitle} filter-ээр overlay хийнэ.
\item Background music сонгогдсон тохиолдолд \code{amix} filter ашиглан narration-той хольж ducking хийнэ.
\item Бүх scene-ийн фрагментийг \code{concat} filter-ээр нэгтгэж нэг бүтэн MP4 болгоно.
\item H.264 video codec, AAC audio codec ашиглан encode хийн, resolution-ийг target платформд тааруулна (YouTube --- 1920$\times$1080 30fps, TikTok --- 1080$\times$1920 30fps).
\end{enumerate}

Хэсэгчилсэн re-render нь зөвхөн \code{dirty=true} scene-уудыг шинэчилж, бусад нь хадгалагдсан фрагментийг ашиглах байдлаар ажилладаг. Энэ нь жишээлбэл 6 scene-ийн нэгийг засахад нийт хугацааг 60 секундээс 12--15 секунд болгож бууруулсан.

\subsection{Cloudinary болон AWS S3 нийлмэл стратеги}

Файлын төрлөөс хамаарч хоёр өөр үүлэн хадгалалтын үйлчилгээг ашигласан. Зургийг Cloudinary-д хадгална, учир нь Cloudinary нь image transformation, optimization, CDN-ийг автоматаар хариуцдаг. Видео болон audio-г AWS S3-д хадгалж CloudFront CDN-ээр хүргэнэ. Хэрэв Cloudinary-д алдаа гарвал S3-руу fallback хийнэ.
